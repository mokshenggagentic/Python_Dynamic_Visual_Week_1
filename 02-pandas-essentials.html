Asking Questions Of Data

Filter & Sort — Find what matters, in the order you want

Filtering keeps only rows matching a condition. Sorting reorders rows by a column. These two operations cover 80% of daily data work.

Industry Scenario
Marketing wants to see only high-value orders (revenue > 60,000), sorted from biggest to smallest, to identify VIP customers.
Watch rows get filtered then reordered

vip_orders.py
# FILTER: rows where revenue > 60,000 high_value = df[df["revenue"] > 60000] # SORT: largest revenue first top_orders = high_value.sort_values( by="revenue", ascending=False ) print(top_orders.head()) # Chain both in one line: top = (df[df.revenue > 60000] .sort_values("revenue", ascending=False))
Filtered & sorted output

💡 Why this matters
A boolean mask like df.revenue > 60000 is the most-used pandas pattern in the world. Combine multiple with & (and) and | (or) to slice data any way you need.

Summarising By Category

groupby — Bucket the data, then aggregate

"What's the total revenue per region?" That's a groupby. You split rows into groups (one per region), then apply a summary function (sum, mean, count) to each group.

Industry Scenario
The CFO asks: "Which region performed best this quarter?" Instead of opening a calculator, you groupby region and sum revenue. Done in 1 line.
Watch rows split into buckets, then summed
Step 0 · Ready
Each row will fly to its region's bucket. Then each bucket's revenues are summed.

region_summary.py
# Total revenue per region region_totals = ( df.groupby("region")["revenue"] .sum() .sort_values(ascending=False) ) print(region_totals) # Multiple aggregations at once: df.groupby("region").agg({ "revenue": ["sum", "mean"], "units": "sum" })
region_totals output
region East 200,000 North 212,000 South 75,000 West 98,000 Name: revenue, dtype: int64 # North wins this quarter.
💡 Why this matters
GroupBy is how every dashboard is built. Sales by region, sessions by device, revenue by month — they're all just df.groupby(category)[metric].sum() behind the scenes.

Reshape The Story

A pivot table takes a long list of records and reshapes it into a grid — like rows for products, columns for regions, and values inside being the sums. Excel users love this; pandas does it in one call.

# Reshape: products as rows, regions as cols grid = df.pivot_table( index="product", columns="region", values="revenue", aggfunc="sum", fill_value=0 ) print(grid) # Add totals on rows AND columns: grid_total = df.pivot_table( index="product", columns="region", values="revenue", aggfunc="sum", margins=True, margins_name="Total" )

💡 Why this matters
Pivot tables turn data into at-a-glance answers. Auditors, finance teams, and product managers all use these grids daily. Pandas removes the manual Excel work.

Joining Two Worlds

merge — Combine two tables on a shared key

Real businesses store data across many tables. Orders in one. Customers in another. merge joins them on a shared column (like customer_id), giving you a unified view.

Industry Scenario
An analyst has an orders table (order_id, customer_id, amount) and a customers table (customer_id, name, city). She needs both joined to see which city had the most revenue.
Watch two tables join on a shared key

join_tables.py
# Two separate tables in your warehouse orders = pd.read_csv("orders.csv") customers = pd.read_csv("customers.csv") # Inner join on customer_id full = orders.merge( customers, on="customer_id", how="inner" # only matched rows ) # Now revenue by city in one go full.groupby("city")["amount"].sum()
After merge + groupby
city Mumbai 1170 Delhi 890 Pune 320 Name: amount, dtype: int64 # C99 was DROPPED because that # customer is not in the customers # table. (inner join behaviour)
💡 Why this matters
Every real dataset is split across multiple tables. merge is the SQL JOIN of pandas. Master how="inner" / "left" / "right" / "outer" and you can rebuild any database query.
